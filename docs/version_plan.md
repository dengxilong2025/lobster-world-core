# 版本路线图（Version Plan）与关键里程碑（Milestones）

> 目标：用软件工程的方式把项目拆成可交付版本（v0.x → v1.0），每个版本再拆里程碑，并给出“进入下一阶段”的质量门槛。  
> 原则：不偏离初心（事件溯源 + 投影 + 仿真）；先做可用闭环，再做体验与内容生产；每个版本都做一次第三方测试（Claude/小蓝）。

## 0) 版本命名规则（建议）
- **v0.x**：快速迭代阶段（玩法/叙事循环仍在打磨），但要求核心链路可回放、可解释、可测试
- **v1.0**：对外稳定版本（API/数据结构稳定，具备基本运营能力）
- 里程碑用 **Mx**（Milestone x）表示；版本发布点用 **tag**（例如 `v0.1.0`）

## 1) v0.1（第一版可试用：后端闭环 + 观战/回放可用）

### v0.1-M1：核心闭环（必须）
- Sim → EventStore → Projection → API 打通
- SSE 可用、replay/export 可用、replay/highlight 可用
- 背压/限速等“长期运行保护”具备雏形

### v0.1-M2：观战爽感（增强）
- spectator/home：headline/hot_events + 世界阶段/摘要（state + recent）
- replay/highlight：trace + cause narrative + 世界阶段/近期/风险(或建议) beats

### v0.1 质量门槛（进入 tag 前）
- `go test ./...` 全绿
- determinism 类集成测试稳定
- 第三方测试（Claude/小蓝）完成一次：给出 P0/P1/P2 问题清单

## 2) v0.2（“像游戏”的雏形：最薄前端壳 + 可部署 + 可被智能体测试）

### v0.2-M1：Web 雏形（最低可玩）
- 一个最薄网页（不是美术重做）：能
  - 选择 world
  - 提交 intent
  - 观战 home（阶段/摘要/热点）
  - 打开 replay/highlight
  - 导出 replay/export
- 建议：`/ui` 支持脚本化参数（`world_id/goal/autoconnect`），便于 20 智能体批测
- 部署到一个稳定地址（或提供 docker compose 一键启动）

### v0.2-M2：智能体体验测试通道
- 为“20 个龙虾智能体”准备稳定的测试入口与脚本化路径
- 输出：每轮测试的回放导出（NDJSON）与关键指标（通过/失败原因）

### v0.2 质量门槛
- 端到端冒烟测试（Playwright 或脚本）可重复运行
- 第三方测试（Claude/小蓝）：重点审计前端/接口稳定性与可观测性缺口

## 3) v0.3（叙事与玩法循环：更丰富规则 + 更强 replay）

### v0.3-M1：叙事模板化与“事件剧本库”
- intent rules 扩展（不靠纯关键词，逐步结构化）
- shock/演化/关系事件更丰富（2~3 个新剧本）

### v0.3-M2：replay/解说质量升级
- beats 更稳定（模板化）
- 更强“因果链”展示（trace 结构与展示一致）

### v0.3 质量门槛
- 回放输出结构版本化（schema 或兼容策略）
- 第三方测试：重点看“叙事一致性/决定论破坏点”

## 4) v0.4（内容与呈现层起步：2D 视觉风格 + 基础音效）

> 说明：大规模 3D/动画/配音/音乐放在 v0.5+；v0.4 先做“统一风格的薄壳”。

### v0.4-M1：UI/UX 体系与视觉规范
- 统一字体/色板/布局/动效原则
- 观战页从“工具感”变“产品感”

### v0.4-M2：基础素材
- 2D 图标/地图式场景元素
- 基础音效（事件提示音）

### v0.4 质量门槛
- 可用性测试（含 20 智能体 + 人类少量测试）
- 第三方测试：重点看 UX 与信息层级（是否更“好看好懂”）

## 5) v0.5（内容生产期：角色/任务/剧情结构 + 更完整运营）

### v0.5-M1：任务/角色系统（最小）
- 明确“任务是什么、如何触发、如何结算、如何写入事件”

### v0.5-M2：官网/宣发体系起步
- landing page（最小版）
- 内容节奏：每周产出可回放“爆点片段”

### v0.5 质量门槛
- 第三方测试：重点看安全/滥用/可扩展性

## 6) v1.0（对外稳定版）
- API/事件结构稳定
- 观测、部署、回归、发布流程完整
- 官网/宣发与内容生产有稳定流水线
