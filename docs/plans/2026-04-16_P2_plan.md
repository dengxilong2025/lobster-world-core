# P2 实施计划（Tick仿真骨架 + 72h冲击调度器）

> 本计划基于：Tick 默认 **5秒/步（B）**；72小时冲击采用 **半随机（B）**。
> 目标：把“占位 intent 执行器”升级为可扩展的权威 Tick 写侧，使事件序列可推理、可回放、可逐步演进到真实蝴蝶效应。

## P2 范围与验收

### P2-M1（必须）：Tick 仿真骨架（写侧）
**验收**
- `POST /api/v0/intents` 不再直接 sleep 发事件；而是提交到仿真引擎队列
- 同一 world_id 内，事件具有单调递增 `tick`
- 集成测试：提交 intent 后在事件流中看到 `intent_accepted` → `action_started` → `action_completed`，且 tick 单调递增

### P2-M2：最小世界状态（抽象指标）
**指标（v0）**
- food/population/order/trust/knowledge/conflict
**验收**
- action_completed 附带 `delta`（至少影响 1–2 个指标）
- `GET /api/v0/world/status?world_id=...` 返回当前指标（读侧可先直接取内存快照，后续再投影化）

### P2-M3：72小时冲击调度器（半随机 B）
**验收**
- 冲击三段式：warning → started → ended（事件类型沿用既有 spec）
- 支持“时间窗口 + 权重抽取 + 冷却期”，避免同类冲击刷屏
- 同一 world 的调度逻辑可配置（后续可切换到固定节目单 A）

### P2-M4：里程碑复盘
- 输出：`docs/retrospectives/YYYY-MM-DD_P2.md`

## 工程约束（不可违背）
- 高内聚低耦合：仿真引擎（sim）不依赖 HTTP；HTTP 只是调用 sim 的接口
- 解耦：事件对象不随传输层改变；读模型可丢可重建
- 测试牵引：每个新行为先写集成测试；提交前 `go test ./...` 必须全绿
